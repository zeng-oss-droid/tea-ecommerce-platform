## 1. 项目分层与请求链路

### 1.1 目录结构（后端）

```
src/main/java/com/reservation/teaecommerceplatform/
├── controller/     # 接收 HTTP，返回 Result
├── service/        # 业务接口
│   └── impl/       # 业务实现（事务、校验）
├── mapper/         # MyBatis 接口
├── entity/         # 与表对应的实体
├── dto/            # 登录、注册等入参
├── common/         # Result 等公共类
├── config/         # WebConfig、Cors、Redis
├── interceptor/    # JwtInterceptor、AdminInterceptor
└── util/           # JwtUtil、Md5Util

src/main/resources/
├── mapper/*.xml    # SQL
├── db/init.sql     # 建表
└── application.properties
```

### 1.2 目录结构（前端）

```
frontend/src/
├── views/          # 页面（Cart、Addresses、OrderDetail…）
├── views/admin/    # 管理/商家共用页面
├── layouts/        # MainLayout、AdminLayout
├── stores/user.js  # Token、用户信息
├── utils/api.js    # Axios + 拦截器
└── router/index.js # 路由与权限守卫
```

### 1.3 一次请求的完整路径

```
浏览器页面
  → api.post('/order/create', body)     // frontend/src/utils/api.js
  → Vite 代理 /api → localhost:8080
  → JwtInterceptor：解析 Token → request.setAttribute("userId", …)
  → OrderController.createOrder()
  → OrderServiceImpl.createOrder()      // @Transactional
  → OrderMapper / OrderItemMapper …     // *.xml 执行 SQL
  → MySQL
  ← JSON：{ "code": 200, "message": "…", "data": { … } }
  ← api 拦截器：code===200 才交给页面使用 res.data
```

**要点**：前端不直连数据库；`userId` 由后端从 JWT 解析，不信任前端传的 userId。

## 2. 统一返回 Result

**文件**：`common/Result.java`

| 字段 | 含义 |
|------|------|
| code | 业务码：200 成功，401 未登录，500 一般错误 |
| message | 提示文字 |
| data | 业务数据（泛型 T） |

常用写法：

```java
return Result.success("订单创建成功", order);
return Result.error(e.getMessage());
return Result.error(401, "未登录或token已过期");
```

**前端配合**（`utils/api.js`）：`res.code === 200` 视为成功，业务数据在 `res.data`；否则 `Promise.reject`，页面 `catch` 后 `ElMessage.error(error.message)`。

## 3. 登录认证与权限

### 3.1 登录发 Token

| 环节 | 位置 |
|------|------|
| 前端表单 | `views/Login.vue` → `userStore.login()` |
| 存 Token | `stores/user.js` → `localStorage` |
| 后端 | `UserController.login` → `UserServiceImpl.login` |
| 密码 | `Md5Util.encrypt` 后与库中比对 |
| 签发 JWT | `JwtUtil.generateToken(userId, username)` |

Token 载荷含 `userId`、`username`，**不含 role**。

### 3.2 请求自动带 Token

`api.js` 请求拦截器：

```javascript
config.headers.Authorization = `Bearer ${userStore.token}`
```

### 3.3 JwtInterceptor（认人，不拦路）

**文件**：`interceptor/JwtInterceptor.java`  
**注册**：`config/WebConfig.java` → 拦截 `/api/**`（排除登录、注册、商品列表等）

逻辑：

1. 读 `Authorization: Bearer xxx`
2. 校验 Token → `request.setAttribute("userId", userId)`
3. **始终 return true**；未登录则 userId 为空，由 Controller 判断

### 3.4 AdminInterceptor（管理员门禁）

拦截：`/api/admin/**`、上传、分类写接口。  
逻辑：Token 有效且 **数据库 `user.role == 1`** 才放行，否则 HTTP 403。

### 3.5 商家识别

- 数据库：`user.role = 2`（入驻审核通过后 `approveMerchantApplication` 更新）
- 前端：`userInfo.role === 2` 显示「商家后台」，路由 `/merchant/*`
- 后端：`MerchantController.checkMerchantRole()` 必须 role=2；用 `merchant.getId()` 作 `sellerId` 过滤本店数据

### 3.6 角色一览

| role | 身份 | 前端入口 | 后端接口前缀 |
|------|------|----------|----------------|
| 0 | 普通用户 | 商城前台 | `/api/order`、`/api/cart` 等 |
| 1 | 管理员 | `/admin` | `/api/admin/**` |
| 2 | 商家 | `/merchant` | `/api/merchant/**` |

## 6. 购物车

### 6.1 流程

```
商品页「加入购物车」
  → POST /api/cart/add { productId, quantity }
  → CartServiceImpl.addToCart
  → 写 cart 表（同用户同商品合并数量）
```

### 6.2 核心逻辑（CartServiceImpl）

1. 查 `product`：是否存在、是否上架、库存够不够  
2. 若购物车已有同款：`updateQuantity` 累加  
3. 否则 `insert` 新行  

**表 `cart`**：`user_id`、`product_id`、`quantity`，唯一约束 `(user_id, product_id)`。

### 6.3 前端页面（Cart.vue）

| 操作 | 接口 |
|------|------|
| 列表 | `GET /api/cart/list` + 并发拉商品详情 |
| 改数量 | `PUT /api/cart/update` |
| 删除 | `DELETE /api/cart/remove/{cartId}` |
| 去结算 | 跳转 `/addresses?from=cart&selectedIds=1,2` |

**文件**：`controller/CartController.java`、`service/impl/CartServiceImpl.java`、`mapper/CartMapper.xml`

### 7.1 两种使用场景

| 场景 | 页面 | 行为 |

### 7.2 地址表单写库

```javascript
// 新增
await api.post('/address/add', form)
// 更新
await api.put('/address/update', { ...form, id })
```

后端：`AddressController` → `AddressService` → `AddressMapper.insert/update`

### 7.3 与订单的关系

下单时只传 **`addressId`**，订单表存 `address_id` 外键，不把地址字符串冗余进订单主表。

## 8. 订单（创建 / 支付 / 取消 / 收货）

### 8.1 订单状态

| status | 含义 |
|--------|------|
| 0 | 待支付 |
| 1 | 已支付 |
| 2 | 已发货 |
| 3 | 已完成 |
| 4 | 已取消 |
| 5 | 退款中 |
| 6 | 已退款 |

### 8.2 创建订单（核心）

**前端**（`Addresses.vue` → `createOrder`）：

```javascript
await api.post('/order/create', {
  addressId: selectedAddressId.value,
  remark: '',
  cartIds: cartItems.value.map(item => item.id)
})
```

**后端入口**：`OrderController.createOrder` → `OrderServiceImpl.createOrder`（**@Transactional**）

| 步骤 | 做什么 | 表 |
|------|--------|-----|
| 1 | 校验地址属于当前用户 | address（读） |
| 2 | 按 cartIds 过滤购物车 | cart（读） |
| 3 | 校验商品、库存，算总金额 | product（读） |
| 4 | 生成订单号，组装 Order | — |
| 5 | insert 订单主表 | **order** |
| 6 | 每条购物车 → insert 订单项 | **order_item** |
| 7 | 扣库存 | **product** |
| 8 | 清空该用户购物车 | **cart**（删） |

**文件**：`service/impl/OrderServiceImpl.java`（约 40～167 行）、`mapper/OrderMapper.xml`、`OrderItemMapper.xml`

### 8.3 支付（模拟）

| 前端 | 后端 |
|------|------|
| `OrderDetail.vue` → `POST /api/order/pay` | `OrderServiceImpl.payOrder`：status 0→1，记 payType、payTime |

### 8.4 取消订单

仅 **待支付(0)** 可取消：恢复库存，status→4（`cancelOrder`）。

### 8.5 确认收货

仅 **已发货(2)** 可确认：status→3，并按订单项数量 **累加商品销量**（`confirmReceipt`）。

### 8.6 相关接口一览

| 接口 | Controller 方法 |
|------|-----------------|
| POST `/api/order/create` | createOrder |
| GET `/api/order/list` | getOrderList |
| GET `/api/order/detail/{id}` | getOrderDetail |
| POST `/api/order/pay` | payOrder |
| POST `/api/order/cancel/{id}` | cancelOrder |
| POST `/api/order/confirm/{id}` | confirmReceipt |

## 9. 退款

### 9.1 申请退款

**前端**：`OrderDetail.vue` → `POST /api/refund/create`  
**后端**：`RefundServiceImpl.createRefund`（@Transactional）

主要逻辑：

1. 订单必须属于当前用户  
2. 订单状态不能是待支付/已取消/退款中/已退款  
3. 不能已有进行中的退款单  
4. insert `refund`，生成 `refundNo`  
5. **订单 status → 5（退款中）**

**表 `refund`**：`type`（仅退款/退货退款）、`reason`、`refund_amount`、`status` 等。

### 9.2 退货快递单号（在用）

用户填写退货物流：`POST /api/refund/update-logistics/{refundId}`  
写入 **`refund.logistics_no`**（不是 `logistics` 表）。

### 9.3 退款状态（简化）

由管理员/商家在后台审核、同意、拒绝、确认收货等（`RefundServiceImpl` 内多段状态判断 + `RefundMapper`）。

**文件**：`RefundController.java`、`RefundServiceImpl.java`、`views/OrderDetail.vue`、`views/admin/RefundManagement.vue`

## 10. 评价（评分）

### 10.1 规则

- 仅订单 **status = 3（已完成）** 可评价  
- 每个 **order_item** 只能评一次  
- `rating`：1～5 星；`content` 选填  
- 提交后 **status = 1（已通过）**，商品页只展示已通过评价  

### 10.2 提交评价

**前端**（`OrderDetail.vue`）：

```javascript
await api.post('/review/create', {
  orderId, orderItemId, productId, rating, content
})
```

**后端**：`ReviewServiceImpl.createReview`  
校验订单 → 防重复 → 从订单项快照商品名/图 → `reviewMapper.insert`

### 10.3 展示评价

| 场景 | 接口 | SQL 要点 |
|------|------|----------|
| 商品详情 | GET `/api/review/product/{id}` | `status=1`，联表 user 昵称头像 |
| 订单详情 | GET `/api/review/order/{id}` | 本单已评列表 |

**表 `review`**：`order_id`、`order_item_id`、`product_id`、`rating`、`content`、`reply`（商家回复）

**文件**：`ReviewController.java`、`ReviewServiceImpl.java`、`ReviewMapper.xml`、`ProductDetail.vue`

## 11. 管理端与商家端

### 11.1 共用布局，按路由区分

`AdminLayout.vue`：

```javascript
const isMerchantOnly = computed(() => route.path.startsWith('/merchant'))
```

- `/admin/*`：管理员菜单（用户、分类、全站订单、商家申请…）  
- `/merchant/*`：商家菜单（本店商品、订单、退款）

### 11.2 同一页面，不同 API

例：`OrderManagement.vue`

```javascript
const endpoint = userStore.userInfo?.role === 2
  ? '/merchant/orders'
  : '/admin/orders'
```

### 11.3 商家数据隔离

`OrderServiceImpl.getMerchantOrders(sellerId, …)`：通过 `order_item.seller_id` 关联，只查包含本店商品的订单。  
改订单状态前：`orderItemMapper.countByOrderIdAndSellerId` 校验权限。

### 11.4 商家入驻

| 前端 | 后端 |
|------|------|
| `JoinUs.vue` 提交申请 | `UserController` 申请接口 |
| `admin/MerchantApplicationManagement.vue` | `AdminController` 审核通过 → `role=2` |
